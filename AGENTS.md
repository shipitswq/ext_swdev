## Agent Team Orchestration

This repository implements a **multi-agent software development team orchestration**.
When starting a new project, the workflow proceeds through sequential phases:

1. **pm** (Product Manager) ‚Äî requirements & PRD
2. **architect** (Architect) ‚Äî system design & interface specs
3. **task-manager** (Task Manager) ‚Äî task decomposition for parallel work
4. **developer** (Developer) ‚Äî code implementation (parallel tasks)
5. **reviewer** (Reviewer) ‚Äî code review per task
6. **integration-manager** (Integration Manager) ‚Äî merge branches & resolve conflicts
7. **tester** (Tester) ‚Äî integration & E2E testing
8. **done** ‚Äî project complete

See `plugins/dev-agent-team/docs/team-workflow.md` for detailed workflow and role responsibilities.

| Command         | Purpose                                |
|-----------------|----------------------------------------|
| `npm run dev`   | Start a development server with HMR    |
| `npm run build` | Build for production                   |
| `npm run test`  | Run all tests                          |
| `npm run lint`  | Lint and format-check all source files |

Substitute `make`, `poetry run`, or `uv run` as appropriate for the project language.

## Coding Style & Naming Conventions

- 2-space indentation for JS/TS; 4 spaces for Python/Rust.
- `PascalCase` for types, classes, components, interfaces. `camelCase` for variables, functions, properties. `kebab-case` for files and directories.
- Run the project formatter (Prettier, Ruff, rustfmt) before committing.
- Treat linter warnings as errors in CI.

## Testing Guidelines

- Target >80% coverage on core logic.
- Use `describe`/`it` blocks that describe behavior in plain English. Co-locate unit tests (`.test.ts` suffix) next to source; place integration tests under `tests/`.
- Run a subset with `npx vitest path/to/test` or `pytest tests/path/ -k "pattern"`.

## Commit & Pull Request Guidelines

- Follow [Conventional Commits](https://www.conventionalcommits.org/): `feat(scope): message`. Types: `feat`, `fix`, `chore`, `refactor`, `docs`, `test`, `style`.
- Keep commits atomic. PRs must describe the change and why, link the related issue, include screenshots for UI changes, and pass CI.


## Auto Project Review

After the Developer phase completes all tasks (MVP ready), the agent **automatically** runs a project review:

1. **Auto project review** ‚Äî Uses `schemas/project-review.md` template, output to `work/project-review.md`
2. **Auto-fix Critical/High** ‚Äî All Critical and High severity issues are automatically fixed, verified by build + tests
3. **User choice** ‚Äî Remaining Medium/Low issues are listed for the user to choose how to proceed (continue, fix selected, or demo first)

See plugins/dev-agent-team/docs/team-workflow.md for details.
## Agent-Specific Instructions

- Prefer the repository's established patterns over new frameworks.
- Read existing config files before assuming the toolchain.
- Do not edit this file unless asked.
- Write comments only where the logic is not self-explanatory.
- When starting a new project, check `project.json` for the current phase, then load the corresponding skill from `plugins/dev-agent-team/skills/`.
- Use `plugins/dev-agent-team/scripts/init-project.ps1` to scaffold new projects, and `plugins/dev-agent-team/scripts/next-phase.ps1` to advance phases.
